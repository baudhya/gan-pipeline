Chapter One

What Is This Project and Why Does It Exist?

Who this is for You are a software developer who has been handed a task: get this GAN pipeline running. You may not know what SAR is, what a GAN is, or what pix2pix does. This chapter explains all three from scratch before you touch a single line of code.

1.1 What is SAR? What is EO?

Satellites orbit the Earth and take pictures. There are two major types of imaging sensors:

SAR — Synthetic Aperture Radar

Emits microwave radio pulses and measures what bounces back. Works through clouds, rain, and darkness. The images look strange to humans — bright areas are rough or metallic surfaces, dark areas are smooth water or pavement.

Satellite: Sentinel-1 (ESA)

EO — Electro-Optical (Optical)

Captures visible light like a camera. Produces natural-looking images humans understand intuitively. But it cannot see through clouds, and it only works in daylight.

Satellite: Sentinel-2 (ESA)

The problem: most of Earth's surface is covered by clouds most of the time. SAR sees everything but is hard to interpret. Optical images are intuitive but often unavailable. The goal of this pipeline is to train a model that takes a SAR image as input and outputs a realistic-looking optical image — giving analysts the best of both worlds.

1.2 What is a GAN?

A Generative Adversarial Network (GAN) is a machine learning architecture consisting of two neural networks that compete against each other:

G
Generator — takes an input and tries to produce a realistic-looking output. It starts knowing nothing and gradually learns to produce convincing images.
D
Discriminator — looks at images and tries to tell whether they are real (from the dataset) or fake (produced by the Generator). It also starts knowing nothing.

They train together in a loop. The Generator gets better at fooling the Discriminator. The Discriminator gets better at spotting fakes. Eventually the Generator produces images realistic enough that the Discriminator cannot reliably tell them apart from real ones. At that point, the Generator is useful.

Analogy Think of the Generator as a counterfeiter and the Discriminator as a detective. The counterfeiter keeps improving their forgeries, the detective keeps improving their detection skills. After enough iterations, the counterfeiter is so good that the detective has to guess randomly — meaning the forgeries are indistinguishable from real currency.

1.3 What does pix2pix do?

pix2pix is a specific GAN architecture for image-to-image translation — transforming one type of image into another. Unlike a basic GAN that generates images from random noise, pix2pix takes a conditioning input (the SAR image) and generates an output conditioned on that input (the corresponding EO image).

It requires paired training data: for every SAR image, you need the matching optical image of the same location at the same time. The Sentinel-1 and Sentinel-2 satellites are both operated by ESA and their images can be co-registered (aligned) to create these pairs.

Training pair: [ SAR image ] ──→ Generator ──→ [ Fake EO image ] │ [ Real EO image ] ─────────────────────┤ ↓ Discriminator (real or fake?)

Configuration System (Hydra)

All hyperparameters — learning rates, batch size, number of layers, loss weights — are stored in YAML files, not hardcoded in Python. The tool that manages this is Hydra.

4.1 Why use Hydra instead of hardcoding?

Hardcoding values like lr = 0.0002 inside Python files means you have to edit source code every time you want to try different settings. Hydra externalises all of these into config files, which means:

  • You can change any setting from the command line without touching Python files
  • Every experiment is reproducible — the config used is saved automatically
  • You can swap entire config groups (e.g. switch from pix2pix to DCGAN) in one flag

4.2 How the config tree is assembled

The root file configs/config.yaml declares which sub-configs to load:

defaults:
  - model: pix2pix        # loads configs/model/pix2pix.yaml
  - training: pix2pix     # loads configs/training/pix2pix.yaml
  - data: sar_eo          # loads configs/data/sar_eo.yaml
  - _self_

experiment_name: sar_eo_pix2pix
seed: 42
output_dir: outputs/${experiment_name}

At runtime, Hydra merges all these files into one big config object that the training script receives.

Data Pipeline — From Raw Satellite to Training Tensors

5.1 Raw Sentinel data

Raw data from ESA comes as GeoTIFF files — multi-channel raster images with geographic metadata. Before training, these must be converted to simple PNGs that PyTorch can load.

The conversion script is scripts/prepare_data.py. It supports two modes:

ModeInputUse when…
sen12msPre-cropped 256×256 GeoTIFF patches from the SEN12MS benchmark datasetYou have SEN12MS downloaded — fastest to get started
scenesFull Sentinel-1 and Sentinel-2 scene GeoTIFFsYou have your own raw satellite data and want to chip it into patches

Example command for SEN12MS:

python scripts/prepare_data.py \
  --mode sen12ms \
  --s1-dir /data/SEN12MS/s1 \
  --s2-dir /data/SEN12MS/s2 \
  --output-dir data/sar_eo \
  --sar-already-db \      # SEN12MS stores SAR already in dB scale
  --sar-channels 1 \      # 1 = VV only (grayscale), 3 = VV+VH+VV (pseudo-RGB)
  --val-split 0.1 \
  --test-split 0.1

5.2 Preprocessing (sentinel_utils.py)

The file src/gan_pipeline/data/sentinel_utils.py contains pure-numpy functions that convert raw sensor values to uint8 images. No rasterio dependency — these can be imported and used anywhere.

SAR preprocessing pipeline

Raw Sentinel-1 GRD backscatter (float32, linear power scale) │ ▼ 10 · log₁₀(max(x, 1e-10)) ← convert to dB. The 1e-10 floor │ prevents log(0) = −∞ ▼ dB values (typically −25 dB to 0 dB for land) │ ▼ clip to [−25, 0] dB ← values below −25 are sensor noise; │ values above 0 are very bright targets ▼ (x − min_db) / (max_db − min_db) ← normalise to [0, 1] │ ▼ × 255 → round → uint8 ← ready to save as PNG

EO preprocessing pipeline

Raw Sentinel-2 L2A reflectance (uint16, values = reflectance × 10000) │ ▼ ÷ 10000 ← convert to physical reflectance [0, ~1] │ ▼ clip to [0, 0.3] ← most land surfaces < 0.3 reflectance; │ snow/clouds saturate — acceptable ▼ ÷ 0.3 ← normalise to [0, 1] │ ▼ × 255 → uint8 ← ready to save as PNG

Output format

Both SAR and EO patches are assembled into a single side-by-side PNG: SAR on the left half, EO on the right half. For 256×256 target images this is a 512×256 file.

┌─────────────────┬─────────────────┐ │ │ │ │ SAR 256×256 │ EO 256×256 │ │ (grayscale or │ (RGB optical) │ │ pseudo-RGB) │ │ │ │ │ └─────────────────┴─────────────────┘ 512 pixels wide × 256 pixels tall

The dataset loader splits this back at load time. Storing them together makes shuffling and splitting the dataset simpler.

5.3 Dataset classes (paired_dataset.py)

Two dataset classes handle loading:

SideBySidePairedDataset

Opens each PNG, splits it at the midpoint horizontally, returns a dict with SAR and EO tensors.

5.4 Data augmentation

Augmentation increases the effective size of the dataset by applying random transformations at load time. The key constraint: the same transformation must be applied to both SAR and EO. If you flip one, you must flip the other — otherwise the pair is no longer aligned.

This is achieved using torchvision.transforms.functional with shared random parameters:

StepWhat happensWhy
1. ResizeBoth images scaled to 286×286Makes the random crop always possible
2. Random cropSame random 256×256 region cut from bothSpatial jitter without misalignment
3. Random horizontal flipSame flip/no-flip decision applied to bothDoubles effective dataset size
4. Normaliseuint8 [0,255] → float [-1,1]Neural networks train better on zero-centred inputs

Augmentation is disabled for validation and test splits to ensure consistent evaluation.

5.5 Weighted Random Sampling

The SEN12MS dataset is not class-balanced. Forest and cropland patches make up ~65% of the data. Water and snow patches are under 2% each. If you train on this distribution as-is, the model learns forest and crops very well and performs poorly on rare classes.

Weighted Random Sampling solves this by upsampling rare classes. Each sample is assigned a weight equal to the inverse frequency of its class. The DataLoader then samples according to these weights with replacement, so every class is seen approximately equally per epoch.

from collections import Counter
from torch.utils.data import WeightedRandomSampler

# land-cover class label for each training image
train_classes = [12, 1, 17, 12, 1, ...]   # loaded from manifest.csv

# weight = total_samples / count_of_this_class
class_counts = Counter(train_classes)
total = sum(class_counts.values())
class_weight = {cls: total / cnt for cls, cnt in class_counts.items()}

# one weight per sample
sample_weights = [class_weight[cls] for cls in train_classes]

sampler = WeightedRandomSampler(
    weights=sample_weights,
    num_samples=len(sample_weights),
    replacement=True,   # same sample can appear multiple times per epoch
)

# Pass sampler instead of shuffle=True
loader = DataLoader(dataset, batch_size=1, sampler=sampler)
Important When using a sampler, you must set shuffle=False in the DataLoader (or not pass it at all). Passing both sampler and shuffle=True raises an error.

Chapter Six

Models — Architecture Deep Dive

6.1 U-Net Generator (models/unet.py)

Class: UNetGenerator(in_channels, out_channels, base_features=64)

Parameters: ~54 million

The generator is an 8-level U-Net. The "U" shape comes from an encoder that progressively shrinks the image, followed by a decoder that progressively restores it. Skip connections pass information directly from each encoder level to its matching decoder level.

SAR input (1 ch, 256×256) │ ┌─────┴──────────────────────────────────────────┐ │ ENCODER — halves spatial dims at each step │ │ │ │ enc1: 256→128 (64 channels) no BatchNorm │──┐ │ enc2: 128→64 (128 ch) │──┤ │ enc3: 64→32 (256 ch) │──┤ │ enc4: 32→16 (512 ch) │──┤ skip │ enc5: 16→8 (512 ch) │──┤ connections │ enc6: 8→4 (512 ch) │──┤ │ enc7: 4→2 (512 ch) │──┤ │ enc8: 2→1 (512 ch) BOTTLENECK │──┤ └──────────────────────────────────────────────────┘ │ │ │ ┌─────┴──────────────────────────────────────────┐ │ │ DECODER — doubles spatial dims at each step │ │ │ │ │ │ dec1: 1→2 + skip from enc7 → 1024 ch │◄───┤ │ dec2: 2→4 + skip from enc6 → 1024 ch │◄───┤ │ dec3: 4→8 + skip from enc5 → 1024 ch │◄───┤ (Dropout 0.5 │ dec4: 8→16 + skip from enc4 → 1024 ch │◄───┤ on dec1-3) │ dec5: 16→32 + skip from enc3 → 512 ch │◄───┤ │ dec6: 32→64 + skip from enc2 → 256 ch │◄───┤ │ dec7: 64→128 + skip from enc1 → 128 ch │◄───┤ │ out: 128→256 → Tanh → (3 ch, 256×256) │◄───┘ └──────────────────────────────────────────────────┘ Fake EO output (3 ch, 256×256, values in [-1, 1])

Why skip connections? The bottleneck (1×1 spatial) forces the network to compress all information into a tiny representation. Low-level details like edges and textures would be lost. Skip connections bypass the bottleneck by sending early-layer features directly to later decoder layers, preserving fine spatial information.

Critical Constraint — Never Change All ReLU activations in the decoder use ReLU(inplace=False). If you change them to inplace=True, PyTorch will raise a RuntimeError: one of the variables needed for gradient computation has been modified by an inplace operation. This is because the skip connections share tensor memory with the encoder, and inplace modification corrupts the values needed for the backward pass.

6.2 PatchGAN Discriminator (models/patchgan.py)

Class: PatchGANDiscriminator(sar_channels, eo_channels, base_features=64, spectral_norm=False)

Instead of outputting a single "real or fake" score for the whole image, PatchGAN outputs a grid of scores — one score per patch of the image. Each score represents "is this patch real or fake?"

Input: concat([SAR, EO or fake_EO]) = (B, 4, 256, 256) │ L1: Conv(4→64, 4×4, stride=2) LeakyReLU no BN → (B, 64, 128, 128) L2: Conv(64→128, 4×4, stride=2) BN LeakyReLU → (B, 128, 64, 64) L3: Conv(128→256,4×4, stride=2) BN LeakyReLU → (B, 256, 32, 32) L4: Conv(256→512,4×4, stride=1) BN LeakyReLU → (B, 512, 31, 31) L5: Conv(512→1, 4×4, stride=1) → (B, 1, 30, 30) │ Output: (B, 1, 30, 30) — 900 independent patch scores Each score covers a 70×70 pixel receptive field in the original image

Why PatchGAN? A full-image discriminator outputs one scalar and gives the generator a weak, diffuse signal. PatchGAN gives 900 independent feedback signals covering every region of the image — the generator gets penalised for any local patch that looks fake, producing sharper textures.

Spectral Normalisation: When enabled (spectral_norm=True), nn.utils.spectral_norm is applied to every Conv2d layer. This constrains the Lipschitz constant of the discriminator, preventing it from becoming too powerful and destabilising training. Applied after weight initialisation so the N(0, 0.02) initialisation is preserved.

6.3 Multi-Scale Discriminator (models/multiscale_disc.py)

Class: MultiScaleDiscriminator(sar_channels, eo_channels, base_features=64, n_scales=3, spectral_norm=False)

Runs N independent PatchGAN discriminators on progressively downsampled versions of the input. Each discriminator has its own weights and evaluates a different aspect of the image quality.

Input pair (B, 4, 256, 256) │ ├──→ D₀ at full res (256×256) → 30×30 patch map ← fine texture, sharp edges │ AvgPool2d(3, stride=2) ← smooth downsampling, no aliasing │ ├──→ D₁ at half res (128×128) → 14×14 patch map ← mid-level structure │ AvgPool2d(3, stride=2) │ └──→ D₂ at quarter res (64×64) → 6×6 patch map ← global layout, composition All three patch maps are collected in a list → losses are averaged across scales

Why multi-scale? A single fine-scale discriminator can miss global inconsistencies — the generator might produce a locally-convincing texture that doesn't make sense globally (e.g. a forest that looks fine patch by patch but has inconsistent tree patterns across the image). The coarse-scale discriminators catch these global failures.

Important implementation detail MultiScaleDiscriminator.forward() returns a list[Tensor], not a single tensor. This is why there are separate multiscale_discriminator_loss() and multiscale_generator_loss() functions in losses.py — they average the loss across all scales.

6.4 DCGAN Models (models/dcgan.py)

Used only in the unconditional pipeline (scripts/train.py). Takes random noise, generates an image — no conditioning on SAR input. Useful for training on datasets like CelebA to verify the architecture and training loop work before tackling the harder SAR→EO task.

Chapter Seven

Loss Functions — What the Model Optimises and Why

The model is trained by minimising a loss function — a number that measures how bad the current output is. Multiple loss terms are combined, each measuring a different quality dimension.

7.1 Adversarial Loss

The adversarial loss is the core GAN signal — it measures how well the Generator is fooling the Discriminator.

Hinge Loss (default — recommended)

L_D = mean(relu(1 - real_logits)) + mean(relu(1 + fake_logits))
L_G = -mean(fake_logits)

Derived from the SVM margin objective. Real samples are rewarded only if their score is above +1; fake samples are penalised only if their score is above −1. This creates a stable "margin" that prevents gradient explosion when the discriminator is very confident.

BCE Loss

L_D = 0.5 × (BCE(real_logits, 1) + BCE(fake_logits, 0))
L_G = BCE(fake_logits, 1)

Standard binary cross-entropy. Simple and well-understood, but can suffer from vanishing gradients when the discriminator becomes very confident early in training.

Wasserstein Loss

L_D = mean(fake_logits) - mean(real_logits)
L_G = -mean(fake_logits)

Measures the "Earth Mover's Distance" between real and generated distributions. Requires gradient penalty (training.gradient_penalty: true) to satisfy the Lipschitz constraint — without it the discriminator can diverge.

7.2 L1 Pixel Loss

L_L1 = mean(|fake_EO - real_EO|)

Measures the average absolute pixel difference between the generated and real image. Weight: λ_L1 = 100 — much higher than the adversarial weight because pixel accuracy is a primary goal.

Why L1 and not L2? L2 (mean squared error) penalises large errors more heavily, which causes the model to produce blurry outputs (the "safe" average of all possible outputs). L1 allows sharper predictions.

7.3 VGG Perceptual Loss

Pixel loss measures low-level accuracy but doesn't capture perceptual quality. Two images can look visually very different but have similar pixel-level L1 distances (e.g. a shifted version of the same image).

The perceptual loss extracts feature maps from a pre-trained VGG16 network at four intermediate layers and measures their L1 distance:

L_VGG = Σ L1(VGG_i(fake_EO), VGG_i(real_EO))   for i in {relu1_2, relu2_2, relu3_3, relu4_3}

Each VGG layer captures increasingly abstract features:

VGG LayerCapturesWhat it encourages
relu1_2Edges, coloursSharp edges, correct colour tones
relu2_2TexturesRealistic surface textures
relu3_3PatternsMid-level structures (roads, field boundaries)
relu4_3SemanticsHigh-level content (building vs vegetation)

The VGG network is frozen (weights don't change during training) — it is purely a feature extractor. Weight: λ_VGG = 10.

7.4 Feature Matching Loss

Introduced in Pix2PixHD. Instead of only using the Discriminator's final output, this loss also uses the Discriminator's intermediate features:

L_FM = mean over (scale s, layer l) of: L1(D_s_l(fake_pair), D_s_l(real_pair).detach())

This pushes the Generator to produce images that activate the Discriminator's internal features in the same way that real images do — a much richer training signal than just the final real/fake score. Weight: λ_FM = 10.

Real features are .detach()-ed so gradients only flow through the fake path to update the Generator.

7.5 Full Generator Loss

L_G_total = L_G_adv ← fool the discriminator + 100.0 × L1(fake_EO, real_EO) ← stay close to real pixels + 10.0 × L_VGG(fake_EO, real_EO) ← match perceptual features + 10.0 × L_FM ← match discriminator internals

Setting any lambda to 0.0 disables that loss term entirely without changing any code:

python scripts/train_pix2pix.py training.lambda_vgg=0.0 training.lambda_fm=0.0

Trainers — The Training Loop Explained

8.1 What a training loop does

Training a neural network is a loop: show the model a batch of data, measure how wrong it is (loss), compute how to improve (gradients via backpropagation), and update the weights (optimiser step). Repeat millions of times.

GANs are more complex because there are two networks (G and D) that must be updated alternately.

8.2 Pix2PixTrainer — one training step

File: src/gan_pipeline/training/pix2pix_trainer.py

For each batch {"sar": ..., "eo": ...}:

1
Generate a fake EO image
fake_eo = generator(sar).detach()
The .detach() prevents gradients from flowing back into the Generator during the Discriminator update — we only want to update D here.
2
Update the Discriminator
Forward the real pair [sar, real_eo] and the fake pair [sar, fake_eo] through D. Compute D loss. Backpropagate. Optimiser step.
3
Generate again (without detach)
fake_eo = generator(sar)
This time we keep the computational graph so gradients can flow back into G.
4
Update the Generator
Forward the fake pair through D. Compute G total loss (adversarial + L1 + VGG + FM). Backpropagate through both D and G. Optimiser step on G only.
5
Log and checkpoint
Log losses to console and MLflow. Save checkpoint and sample images at configured intervals.

8.3 Checkpoint format

Checkpoints are saved as .pt files in outputs/<experiment_name>/checkpoints/:

{
    "epoch": int,
    "generator": OrderedDict,       # generator.state_dict()
    "discriminator": OrderedDict,   # discriminator.state_dict()
    "opt_g": dict,                  # generator optimiser state
    "opt_d": dict,                  # discriminator optimiser state

How to Build a Custom Dataset

This chapter explains how to implement a new dataset class from scratch — for example, if you have a different data format than side-by-side PNGs.

17.1 How PyTorch datasets work

A PyTorch Dataset is any class that implements two methods:

  • __len__(self) — returns the total number of samples
  • __getitem__(self, idx) — returns one sample given its index

The DataLoader wraps a dataset and handles batching, shuffling, and multi-process loading automatically.

17.2 Anatomy of SideBySidePairedDataset

class SideBySidePairedDataset(Dataset[dict[str, torch.Tensor]]):

    def __init__(
        self,
        root: str,
        split: str,           # "train" | "val" | "test"
        image_size: int,
        sar_channels: int,    # 1 = grayscale, 3 = pseudo-RGB
        eo_channels: int,
        augment: bool,
    ) -> None:
        # Build a list of all PNG file paths in root/split/
        self.files = sorted(Path(root, split).glob("*.png"))
        self.image_size = image_size
        self.sar_channels = sar_channels
        self.augment = augment

    def __len__(self) -> int:
        return len(self.files)   # total number of image pairs

    def __getitem__(self, idx: int) -> dict[str, torch.Tensor]:
        img = Image.open(self.files[idx]).convert("RGB")

        # Split at the horizontal midpoint
        w, h = img.size
        sar_img = img.crop((0, 0, w // 2, h))    # left half = SAR
        eo_img  = img.crop((w // 2, 0, w, h))    # right half = EO

        # Apply the same augmentation to both
        sar_img, eo_img = self._augment(sar_img, eo_img)

        # Convert to tensors in [-1, 1]
        return {
            "sar": self._to_tensor_normalized(sar_img, self.sar_channels),
            "eo":  self._to_tensor_normalized(eo_img, 3),
        }

17.3 Implementing your own dataset class

Suppose you have SAR and EO in HDF5 files instead of PNGs. Here is how to implement the dataset:

import h5py
import torch
from torch.utils.data import Dataset

class HDF5PairedDataset(Dataset[dict[str, torch.Tensor]]):

    def __init__(self, h5_path: str, split: str, image_size: int) -> None:
        self.h5_path = h5_path
        self.split = split
        self.image_size = image_size
        # Count samples without loading data into memory
        with h5py.File(h5_path, "r") as f:
            self.length = len(f[split]["sar"])

    def __len__(self) -> int:
        return self.length

    def __getitem__(self, idx: int) -> dict[str, torch.Tensor]:
        # Open the file per call — h5py is thread-safe for reading
        with h5py.File(self.h5_path, "r") as f:
            sar = f[self.split]["sar"][idx]   # numpy array
            eo  = f[self.split]["eo"][idx]

        # Normalise to [-1, 1] and convert to tensor
        sar_t = torch.from_numpy(sar).float() / 127.5 - 1.0
        eo_t  = torch.from_numpy(eo).float()  / 127.5 - 1.0
        return {"sar": sar_t, "eo": eo_t}

17.5 Synchronized augmentation — the key pattern

When augmenting paired data, both images must receive identical spatial transforms. The pattern used in this codebase:

import torchvision.transforms.functional as TF
import random

def apply_paired_augmentation(
    sar: Image.Image, eo: Image.Image, image_size: int
) -> tuple[Image.Image, Image.Image]:
    # Step 1 — resize both to slightly larger than target
    resize_to = int(image_size * 1.12)
    sar = TF.resize(sar, [resize_to, resize_to])
    eo  = TF.resize(eo,  [resize_to, resize_to])

    # Step 2 — compute ONE random crop, apply to BOTH
    i, j, h, w = transforms.RandomCrop.get_params(
        sar, output_size=(image_size, image_size)
    )
    sar = TF.crop(sar, i, j, h, w)
    eo  = TF.crop(eo,  i, j, h, w)

    # Step 3 — ONE flip decision, applied to BOTH
    if random.random() > 0.5:
        sar = TF.hflip(sar)
        eo  = TF.hflip(eo)

    return sar, eo

How to Build a Custom Model

This chapter walks through building a new generator from scratch, registering it in the config system, and integrating it with the training pipeline.

18.1 The base class contract

Every model inherits from BaseGenerator or BaseDiscriminator in models/base.py. The contract:

from abc import ABC, abstractmethod
import torch
import torch.nn as nn

class BaseGenerator(nn.Module, ABC):

    @abstractmethod
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        ...  # subclass must implement this

    def sample(self, n: int, device: torch.device) -> torch.Tensor:
        # default: generate n samples from random noise
        z = torch.randn(n, self.latent_dim, device=device)
        return self(z)  # calls forward(z)

18.2 Building a ResNet Generator step by step

ResNet generators (used in CycleGAN) use residual blocks instead of an encoder-decoder structure. Here is how to build one:

Step 1 — Define a residual block

class ResBlock(nn.Module):
    """A single residual block: Conv→BN→ReLU→Conv→BN with skip add."""

    def __init__(self, channels: int) -> None:
        super().__init__()
        self.block = nn.Sequential(
            nn.ReflectionPad2d(1),          # padding that doesn't create edge artefacts
            nn.Conv2d(channels, channels, 3),
            nn.InstanceNorm2d(channels),    # InstanceNorm works better than BN for style
            nn.ReLU(inplace=False),         # inplace=False — always safe
            nn.ReflectionPad2d(1),
            nn.Conv2d(channels, channels, 3),
            nn.InstanceNorm2d(channels),
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return x + self.block(x)  # residual connection: output = input + learned delta

Step 2 — Build the full generator

class ResNetGenerator(BaseGenerator):
    """ResNet generator: downsample → N residual blocks → upsample."""

    def __init__(
        self,
        in_channels: int = 1,
        out_channels: int = 3,
        base_features: int = 64,
        n_blocks: int = 9,       # 9 for 256×256, 6 for 128×128
    ) -> None:
        super().__init__()

        # Encoder — downsample to smaller spatial size
        encoder = [
            nn.ReflectionPad2d(3),
            nn.Conv2d(in_channels, base_features, 7),
            nn.InstanceNorm2d(base_features),
            nn.ReLU(inplace=False),
            # Two downsampling steps: 256→128→64
            nn.Conv2d(base_features, base_features * 2, 3, stride=2, padding=1),
            nn.InstanceNorm2d(base_features * 2),
            nn.ReLU(inplace=False),
            nn.Conv2d(base_features * 2, base_features * 4, 3, stride=2, padding=1),
            nn.InstanceNorm2d(base_features * 4),
            nn.ReLU(inplace=False),
        ]

        # Residual blocks — process at the compressed resolution
        res_blocks = [ResBlock(base_features * 4) for _ in range(n_blocks)]

        # Decoder — upsample back to original resolution
        decoder = [
            nn.ConvTranspose2d(base_features * 4, base_features * 2, 3, stride=2,
                               padding=1, output_padding=1),
            nn.InstanceNorm2d(base_features * 2),
            nn.ReLU(inplace=False),
            nn.ConvTranspose2d(base_features * 2, base_features, 3, stride=2,
                               padding=1, output_padding=1),
            nn.InstanceNorm2d(base_features),
            nn.ReLU(inplace=False),
            nn.ReflectionPad2d(3),
            nn.Conv2d(base_features, out_channels, 7),
            nn.Tanh(),
        ]

        self.model = nn.Sequential(*encoder, *res_blocks, *decoder)
        self._init_weights()

    def _init_weights(self) -> None:
        for m in self.modules():
            if isinstance(m, nn.Conv2d) or isinstance(m, nn.ConvTranspose2d):
                nn.init.normal_(m.weight, 0.0, 0.02)
                if m.bias is not None:
                    nn.init.zeros_(m.bias)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.model(x)  # type: ignore[no-any-return]

20.2 Building a CycleGAN trainer (unpaired)

CycleGAN trains on unpaired images — you don't need matching SAR/EO pairs, just a set of SAR images and a set of EO images. It trains two generators: G_AB (SAR→EO) and G_BA (EO→SAR), enforcing a cycle consistency: G_BA(G_AB(sar)) ≈ sar.

Step 1 — Create the trainer file

# src/gan_pipeline/training/cyclegan_trainer.py

class CycleGANTrainer:

    def __init__(self, cfg: DictConfig, device: torch.device) -> None:
        # Two generators, two discriminators
        self.G_AB = ResNetGenerator(...).to(device)  # SAR → EO
        self.G_BA = ResNetGenerator(...).to(device)  # EO → SAR
        self.D_A  = PatchGANDiscriminator(...).to(device)  # scores SAR
        self.D_B  = PatchGANDiscriminator(...).to(device)  # scores EO

        self.lambda_cycle = cfg.training.lambda_cycle   # e.g. 10.0
        self.lambda_idt   = cfg.training.lambda_identity  # e.g. 5.0

        # One optimiser per network pair
        self.opt_G = torch.optim.Adam(
            list(self.G_AB.parameters()) + list(self.G_BA.parameters()),
            lr=cfg.training.lr_generator,
            betas=(cfg.training.beta1, 0.999),
        )
        self.opt_D = torch.optim.Adam(
            list(self.D_A.parameters()) + list(self.D_B.parameters()),
            lr=cfg.training.lr_discriminator,
            betas=(cfg.training.beta1, 0.999),
        )

    def _generator_step(self, real_sar, real_eo):
        # Forward cycle: SAR → fake EO → reconstructed SAR
        fake_eo  = self.G_AB(real_sar)
        rec_sar  = self.G_BA(fake_eo)

        # Backward cycle: EO → fake SAR → reconstructed EO
        fake_sar = self.G_BA(real_eo)
        rec_eo   = self.G_AB(fake_sar)

        # Adversarial: G_AB should fool D_B, G_BA should fool D_A
        loss_adv = (
            generator_loss(self.D_B(fake_eo),  LossType.HINGE)
            + generator_loss(self.D_A(fake_sar), LossType.HINGE)
        )

        # Cycle consistency: reconstructed images should match originals
        loss_cycle = (
            F.l1_loss(rec_sar, real_sar)
            + F.l1_loss(rec_eo, real_eo)
        ) * self.lambda_cycle

        # Identity: G_AB(real_eo) should ≈ real_eo (colour preservation)
        loss_idt = (
            F.l1_loss(self.G_AB(real_eo), real_eo)
            + F.l1_loss(self.G_BA(real_sar), real_sar)
        ) * self.lambda_idt

        loss_G = loss_adv + loss_cycle + loss_idt
        self.opt_G.zero_grad()
        loss_G.backward()
        self.opt_G.step()
        return loss_G.item()

Remove a dependency

uv remove scipy               # removes from dependencies + uninstalls
uv remove --dev pytest-cov    # removes from dev group

Upgrade a dependency

uv replaces both pip and venv You do not need python -m venv .venv or pip install when using uv. It manages the virtual environment automatically. The Makefile's make venv and make install targets can be replaced by uv sync.

23.2 Poetry — the most widely used tool

Poetry has been the industry standard for Python project management since 2018. It manages pyproject.toml, virtual environments, and publishing.

Install Poetry

Initialise a new project

Add dependencies

Poetry vs pyproject.toml standard Poetry uses its own [tool.poetry] section in pyproject.toml in addition to the standard [project] section. This means a Poetry-managed pyproject.toml is slightly different from one managed by uv or setuptools. If you use Poetry, stick with it consistently across the project.

23.3 pip-tools — lightweight pinning without a full package manager

If you want to keep plain pip but get reproducible builds with pinned versions, pip-tools is the answer. It compiles pyproject.toml into a locked requirements.txt.

Install pip-tools

pip install pip-tools

Compile requirements from pyproject.toml

# Generates requirements.txt with pinned versions from [project].dependencies
pip-compile pyproject.toml -o requirements.txt

# Generates dev-requirements.txt from [project.optional-dependencies].dev
pip-compile pyproject.toml --extra dev -o requirements-dev.txt

Install pinned requirements

pip-sync requirements-dev.txt   # installs exactly what's in the lock file

Upgrade a pinned package

pip-compile --upgrade-package torch pyproject.toml -o requirements.txt
When to use pip-tools Use pip-tools when you want to keep pip as your installer but need reproducible builds (same exact versions on every machine). It is lighter than Poetry or uv but does not manage virtual environments — you still create and activate the venv manually.

Managing CI/CD from the Command Line

The Goal Every CI/CD action — trigger a workflow, check run status, view logs, manage secrets, test workflows locally — can be done from the terminal without opening a browser. This chapter covers the three tools that make this possible: the GitHub CLI (gh), act (local Actions runner), and the pre-commit CLI.

24.1 GitHub CLI — gh

gh is GitHub's official command-line interface. It communicates directly with the GitHub API, so every action you would take in the web UI can be scripted.

Install

# Ubuntu / Debian
sudo apt install gh

# macOS
brew install gh

# Windows (winget)
winget install GitHub.cli

# Authenticate (one-time)
gh auth login        # prompts: GitHub.com → HTTPS → browser/token

View workflow runs

# List the most recent runs across all workflows
gh run list

# Filter by workflow file
gh run list --workflow pre-commit.yml
gh run list --workflow pytest.yml

# Show status of a specific run (replace with run ID from list)
gh run view 12345678

# Watch a run in real time (streams logs until completion)
gh run watch 12345678

View logs

# Full logs for a run
gh run view 12345678 --log

# Only failed step logs
gh run view 12345678 --log-failed

Trigger a workflow manually

# Run a workflow on demand (requires workflow_dispatch trigger in the YAML)
gh workflow run pre-commit.yml
gh workflow run pytest.yml --ref master

# Pass inputs to a parametrised workflow
gh workflow run train.yml --field epochs=50 --field loss_type=hinge

To enable manual triggering, add workflow_dispatch: to your workflow YAML:

on:
  push:
    branches: [master]
  pull_request:
    branches: [master]
  workflow_dispatch:          # ← adds "Run workflow" button + gh workflow run support
    inputs:
      epochs:
        description: Number of training epochs
        default: "10"

Re-run a failed workflow

# Re-run all jobs in a failed run
gh run rerun 12345678

# Re-run only failed jobs (skips passing jobs — faster)
gh run rerun 12345678 --failed

Download workflow artifacts

# List artifacts produced by a run
gh run download 12345678 --dry-run

# Download all artifacts into ./artifacts/
gh run download 12345678 --dir artifacts/

# Download a specific named artifact
gh run download 12345678 --name coverage-report

Manage secrets from the CLI

# Set a repository secret (prompts for value securely)
gh secret set WANDB_API_KEY
gh secret set HF_TOKEN --body "hf_abc123xyz"

# List secrets (names only — values are never shown)
gh secret list

# Delete a secret
gh secret delete WANDB_API_KEY
Secrets in this project This project currently needs no secrets for CI — tests run on CPU with synthetic data. If you add cloud storage (S3, GCS) or experiment tracking (W&B), add the keys with gh secret set and reference them in the workflow YAML as ${{ secrets.KEY_NAME }}.

Manage workflow files

# List all workflows in the repo
gh workflow list

# Disable a workflow (stops it from triggering)
gh workflow disable pytest.yml

# Re-enable it
gh workflow enable pytest.yml

# View the YAML of a workflow without opening the file
gh workflow view pre-commit.yml

Open the run in a browser (when you need the visual diff)

gh run view 12345678 --web     # opens in browser
gh repo view --web             # open the repo home page

24.2 act — Run GitHub Actions Locally

act is an open-source tool that runs GitHub Actions workflows on your machine using Docker. It lets you test workflow changes without pushing to GitHub and waiting for a runner.

Install act

# macOS
brew install act

# Linux (via release binary)
curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash

# or via Go
go install github.com/nektos/act@latest

act requires Docker to be running. It pulls GitHub-runner-compatible Docker images on first use.

Run all workflows locally

# Simulate a push event (what runs on git push)
act push

# Simulate a pull_request event
act pull_request

# Run a specific workflow
act push --workflows .github/workflows/pytest.yml

# Run a specific job inside a workflow
act push --job run-tests

Choose the runner image

# Micro image — fast but minimal (good for pure-Python, no Docker-in-Docker)
act push -P ubuntu-latest=catthehacker/ubuntu:act-latest

# Full image — matches GitHub's hosted runner (large, ~18 GB)
act push -P ubuntu-latest=ghcr.io/catthehacker/ubuntu:full-latest

For this project the micro image is sufficient — all CI needs is Python and pip.

Pass secrets to act

# Inline
act push --secret WANDB_API_KEY=abc123

# From a file (one KEY=value per line, like .env)
act push --secret-file .secrets

Dry run — see what would execute without running

act push --dryrun

Typical act workflow for this project

# 1. Edit a workflow file
vim .github/workflows/pytest.yml

# 2. Test it locally before pushing
act push --workflows .github/workflows/pytest.yml -P ubuntu-latest=catthehacker/ubuntu:act-latest

# 3. If it passes locally, push
git add .github/workflows/pytest.yml
git commit -m "Update pytest workflow"
git push
act limitations
  • Services (Postgres, Redis) defined under services: may not start correctly in all images
  • Actions that use GITHUB_TOKEN for write operations (create release, deploy) need --actor and a real token
  • GPU runners are not available locally through act — skip GPU-dependent jobs with an if: condition during local testing

24.3 Pre-commit CLI — Full Control Over Hooks

Pre-commit has a rich CLI beyond pre-commit install. You can run individual hooks, update hook versions, and audit what hooks are configured — all without committing.

Run all hooks against all files

pre-commit run --all-files

Run a specific hook only

pre-commit run ruff --all-files
pre-commit run black --all-files
pre-commit run mypy --all-files
pre-commit run pytest --all-files

Run hooks against staged files only (same as what git commit triggers)

pre-commit run

Update hook versions to latest

# Update all hooks to their latest tagged versions
pre-commit autoupdate

# The .pre-commit-config.yaml rev: fields are rewritten automatically
# Review the diff, then commit:
git diff .pre-commit-config.yaml
git add .pre-commit-config.yaml
git commit -m "Update pre-commit hook versions"

Show installed hooks and their status

pre-commit run --show-diff-on-failure --all-files

Clean the pre-commit cache (if hooks behave strangely)

pre-commit clean         # removes cached hook environments
pre-commit run --all-files  # re-installs and runs

Temporarily skip a specific hook

# Skip one hook by its id
SKIP=mypy git commit -m "wip: skip typecheck this once"

# Skip multiple hooks
SKIP=mypy,pytest git commit -m "wip: skip slow checks"
Project policy --no-verify skips ALL hooks and is forbidden by this project's /commit skill. Use SKIP=hook-id when you genuinely need to bypass one hook — it is explicit and auditable.

Uninstall / reinstall hooks

pre-commit uninstall      # removes the git hook scripts
pre-commit install        # reinstalls them from .pre-commit-config.yaml

24.4 Workflow configuration reference for this project

The two workflow files live in .github/workflows/. Here is what each flag does:

# .github/workflows/pre-commit.yml
name: pre-commit
on:
  push:
    branches: [master]       # runs on every push to master
  pull_request:
    branches: [master]       # runs on every PR targeting master

jobs:
  pre-commit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.10"      # pins the Python version
          cache: pip                  # caches pip downloads between runs
      - name: Install deps
        run: pip install -e ".[dev]"  # installs the package + dev extras
      - name: Run pre-commit
        run: pre-commit run --all-files  # runs all 5 hooks
# .github/workflows/pytest.yml
name: pytest
on:
  push:
    branches: [master]
  pull_request:
    branches: [master]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.10"
          cache: pip
      - name: Install deps
        run: pip install -e ".[dev]"
      - name: Run tests
        run: pytest --no-header -q    # same flags as the pre-commit hook

Add a caching layer for faster runs

      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('pyproject.toml') }}
          restore-keys: |
            ${{ runner.os }}-pip-

Insert this step before Install deps. The cache key changes whenever pyproject.toml changes, so stale environments are never reused.

Add a test matrix (test multiple Python versions)

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
    steps:
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
